# rust-development-pipeline

用大模型做规划，把规划编译成确定性脚本，让任何模型都能稳定执行，顺便省省 token , 力图做到“好钢花在刀刃上”。

---

## 起因

刚上手使用 Claude Code 的时候，因为以前对“Agent 协作”这回事有所耳闻，就按照角色分工创了这四个 Agent：`rust-architect`、`strict-code-reviewer`、`plan-decomposer`、`implementation-executor`, 负责规划、核对、分解步骤和执行。 后来做了技能来省去手动敲提示词的麻烦，`/next-phase-plan`、`/implementation-executor`、`/review-pr`。这时候还是以 Agent 为中心—— skill 存在的目的就是更高效地调用 Agent 。后来发现这样 token 烧太快，刚好我新买了台 24GB 内存的 M4 Mac Mini ，就摸索起多大的模型能跑，同时还能在 Claude Code 上完成任务。  
这时发现，运行 Claude Code 和部署一个聊天机器人完全不是一回事。Claude Code 每次对话的第一条消息就会带着长长的 system prompt 直接发出 10-20K token 。小模型自身要 5-7GB，上下文在 agentic 编程会话里还要再吃 5-10GB，最后多番实验下来，在我的 24GB M4 Mac Mini 上，能塞进去的就不超过 9B。更小的模型，真干活就容易变成无头苍蝇。
为了不让本地模型抓瞎犯傻，我在官方推出 Advisor mode 之前就想出了这样一个方案: Opus 写完指令，Haiku 先过一遍，标出看不懂的地方，decomposer 返工，如此迭代到没有问题，再交给本地模型执行。
即便如此，随着执行模型的智力下降，错误和不听使唤的情况屡屡发生。我也迅速体会到大模型驱动的代码编辑与生成其实会有两种潜在的出错方式：

1. 同样的指令换个模型跑，甚至同样的模型再来一遍，结果都不一样。
2. 模型读错了你的意图，写出来的东西看着对，实际上总有些地方没有按照你给的规范来。

这两种失败都不会报错，流水线照样汇报成功。这都归结于 LLM 的"抽奖"本质。

一开始，我按照 harness engineering 的说法，努力改进 skill , 想用规范化的流程和提示，让能力有限的模型只干杂活就好了。起初还算见效，但随着 skill 文件里的规则和边界情况越堆越多，自然语言指令就开始超出 9B 模型的理解范围了，九成的错误都是小模型自己跑偏，不是 API 模型的规划写得有问题。
我起初觉得，反正不是要紧项目，就只是忍着，跑乱了就 `git reset` 让模型再来一轮，指望它这次能聪明一点。好在有的是时间折腾。有一天开始研究 Claude Code 的 hooks，一开始只是想把权限请求转发到 Discord，这样我离开电脑休息的时候也能处理。做着做着就突然想到：hooks 可以跑脚本，它们运行在上下文窗口外面，有自己的进程生命周期，I/O 是确定性的，这就让它成了最强的约束，因为行为是可预测的。

我忽然意识到：其实大部分要么是改已有的行（就是替换），要么是写新文件，两种都可以"编译"成脚本操作。于是，我的流水线就变成了：

```
旧路径：  读规划 → 模型理解 → 模型写代码 → 验证
                          ↑ 模型在这里搞丢

新路径：  读规划 → 规划编译成脚本 → 模型执行脚本 → 验证
                          ↑ 模型只管执行，不需要理解
```

小参数的 LLM 就这样被逼着执行，没有了自己发挥解读的空间，只作为一个流程控制器 (orchestrator)：下一个任务是什么、验收命令过了没、继续还是重试，这些都由 Claude Code 的 hook 事件和脚本来保证。

在 `castep_workflow_framework` 上做了测试，那时候仓库还没创建，跑了多个阶段，同样的问题反复冒出来：模块连接留了口子、单 crate 检查抓不到跨 crate 的问题、审查发现的毛病在规划阶段就该拦住。修复任务和实施任务比例 1.0x 到 3.3x，每做一个实施任务，后面就拖着 1-3 个修复任务。

这些问题推出了 v2.0 和 v2.1，然后创了仓库来打包发布。

## 留下来的原则

| 原则                          | 从哪来的                                                   |
| ----------------------------- | ---------------------------------------------------------- |
| 规划和执行分开                | 小模型跑崩了 Opus 生成的指令                               |
| 编译脚本是唯一执行路径        | 自然语言技能对 9B 模型太复杂                               |
| hooks 强制阶段边界            | hooks 跑在 LLM 外面，不依赖模型能力                        |
| `before` 内容是地址，不是行号 | 前面的任务会改变内容；字节级匹配是唯一可靠的地址           |
| 先诊断再重试                  | 对同一个失败脚本重试三次没用——模型根本诊断不了哪里出了问题 |
| 从源头预防，事后兜底          | 每轮修复花 10+ 次智能体调用；预防省得多                    |
| 范围分类                      | `[Improvement]` 项写到 `deferred.md`，不进修复规划         |

## 怎么跑

### 流水线

```
/next-phase-plan          → 讨论目标 → PHASE_PLAN.md
/plan-review              → 验证架构
/enrich-phase-plan        → architect → decomposer → 试运行编译 → phase-X.Y.toml
/compile-plan             → TOML → compiled/*.sh（sd -F 脚本）
/implementation-executor  → 逐个任务执行脚本
/review-pr                → 四轴审查 → fix-plan.toml + deferred.md
/fix                      → 确定性修复
```

### 各部分

**指令审查** — 任务描述在规划定稿之前会过一遍 Haiku。Opus 负责写，Haiku 来读，标出任何表述不清或有歧义的地方，decomposer 再返工。用小模型来审查给小模型用的指令，这正是关键——Haiku 能发现 Opus 读自己输出时注意不到的问题。

**TOML 规划**用精确的 `before`/`after` 字符串对，执行时不需要任何解释。`before` 字段是内容寻址而不是行号，所以前面的编辑不会让它失效。

**编译 `sd` 脚本**（`sd -F <before> <after> <file>`）做精确字符串替换，LLM 不需要把指令理解成代码变更，只管流程控制。

**PostToolUse hook**（`post_compiled_script.py`）在子智能体执行完编译脚本后让它停下，控制权回到编排器，子智能体不会继续自己理解接下来要做什么。

**SubagentStop hook**（`verify_impl_task.py`）执行验证命令，更新检查点，每个任务后自动提交，包含工作区级 `cargo check --workspace` 门控做跨 crate 验证。

**检查点/恢复**把任务状态写到磁盘，流水线中途崩溃就从最后一个好的任务接着跑，不需要重做。

**范围分类**把每个审查问题标记为 `[Defect]`、`[Correctness]` 或 `[Improvement]`，只有前两个进修复规划，`[Improvement]` 项写到 `deferred.md` 留到下个阶段再想。

**完整性包络**要求每个任务执行完后代码库保持可编译、可访问。"创建文件" + "连接模块" + "更新消费者"是一个任务，不是三个，这防止了最常见的反复失败：不可达的死代码。

**试运行编译**把完整 TOML 规划应用到临时 worktree，跑 `cargo check --workspace`，在执行前把错误反馈给 decomposer，最多两轮修订。

**跨轮模式提取**从之前的 `fix-plan.toml` 里提取反复出现的失败类型（连接遗漏、过期引用、类型不匹配），作为 `KNOWN_FAILURE_MODES` 传给 decomposer，主动检查，不是事后补救。

## 目录结构

```
rust-development-pipeline/
├── agents/                    # 智能体定义
├── hooks/                     # PostToolUse & SubagentStop hooks
├── scripts/                   # sidecar 工具
├── skills/                    # 斜杠命令

```

## 为什么存在

要是硬件够用，这个项目大概不会存在。24GB 内存的限制逼我走了一条原来根本不会想到的路：LLM 控制流程，写代码是脚本的活。

规划需要推理质量，执行需要确定性，把这两件事分开，执行阶段就能跑在任何能塞进去的廉价小模型上，规划阶段留给大模型。

流水线也从错误里学习。`deferred.md` 跨审查轮次积累超出范围的改进建议，不让它们丢掉；跨轮模式提取从之前的修复规划里提取反复出现的失败模式，提前交给 decomposer。每轮修复要花掉 10+ 次智能体调用，在规划阶段预防，省的是好几个数量级。

LLM 上下文的每个字节都留给推理，不是文件 I/O。
